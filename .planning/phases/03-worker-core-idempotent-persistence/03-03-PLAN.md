1. Add a `services:` block to the `verify` job (peer of `runs-on`/`steps`), using images matching docker-compose. Publish Postgres on 5432 and Redis on 6379 (CI host ports), with health checks:
```yaml
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_USER: omnisync
          POSTGRES_PASSWORD: omnisync
          POSTGRES_DB: omnisync
        ports:
          - 5432:5432
        options: >-
          --health-cmd "pg_isready -U omnisync"
          --health-interval 5s
          --health-timeout 3s
          --health-retries 10
      redis:
        image: redis:7
        ports:
          - 6379:6379
        options: >-
          --health-cmd "redis-cli ping"
          --health-interval 5s
          --health-timeout 3s
          --health-retries 10
```

2. Add a job-level `env:` block so all steps (and vitest.setup.ts via `?? process.env`) point at the service containers. CI publishes Postgres on 5432 (NOT the local 5433):
```yaml
    env:
      DATABASE_URL: postgresql://omnisync:omnisync@localhost:5432/omnisync
      DIRECT_URL: postgresql://omnisync:omnisync@localhost:5432/omnisync
      REDIS_URL: redis://localhost:6379
```

3. Add a step that applies migrations to the service DB AFTER "Generate Prisma client" and BEFORE "Test (with coverage gate)":
```yaml
      - name: Apply database migrations
        run: pnpm --filter @omnisync/db exec prisma migrate deploy
```
(`prisma migrate deploy` is the non-interactive CI variant; it reads DATABASE_URL from the job env via prisma.config.ts.)

Keep all existing steps (checkout, pnpm setup, node, install, generate, typecheck, build, test --coverage, lint) in order. The existing "Test (with coverage gate)" step (`pnpm test -- --coverage`) now also runs the worker + db integration suites against the live service containers.
