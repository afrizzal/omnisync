# Plan 04-06: Integration proof — DLQ Postgres mirror + re-queue idempotency

## Objective

Prove the two requirements that mocks cannot verify: RES-03 (the DLQ entry lands durably in Postgres `dlq_events`) and RES-06 (re-queue reprocesses idempotently — exactly one `events` row even on double re-queue). These run against the real local docker-compose Postgres + Redis (the same harness Phase 3 integration tests use, mirrored in CI service containers). After this plan the full worker suite must be green and coverage must stay ≥80% (the gate is already wired in `vitest.config.ts`).

This plan adds tests only — no source changes. It is the Nyquist sign-off for the phase's integration-level requirements.

(In CI these are set by the workflow `env:` block to the service-container values — the `??` lets CI win, mirroring the worker setup pattern.)

Test flow:
1. Build real `createPrismaClient({ max: 5 })`, `createRedisConnection(env.REDIS_URL)`, `createEventsQueue(connection)`, and a worker via `buildWorker({ prisma, connection, logger: noop, crmClient: { sync: async () => {} }, crmPolicy: createCrmPolicy(10000), ttlMs: 30000 }, 5)`. (Inject a no-op CRM client so the pipeline completes without a real mock-crm.)
2. Seed a `dlq_events` row directly: `prisma.deadLetterEvent.create({ data: { fingerprint, source: "SHOPEE", eventType: "order.created", payload: { source:"SHOPEE", eventType:"order.created", externalId:"ext-rq-1", occurredAt:"2026-01-01T00:00:00.000Z", payload:{} }, failureReason: "seed", attempts: 5 } })` with a unique 64-hex fingerprint.
3. Call `requeueDlqEntry({ prisma, queue }, dlqRow.id)` → assert result `{ status: "requeued", fingerprint }`.
4. Bounded-poll `prisma.event.count({ where: { fingerprint } })` until 1 (max 10 * 500ms) → assert `=== 1`.
5. IDEMPOTENCY: call `requeueDlqEntry(...)` a SECOND time on the same id. It will return `not_found` (the row was marked resolved/the entry still exists — adjust: if your service marks resolved, the second findUnique still returns it; re-add with the same jobId is deduped by BullMQ). Either way, bounded-poll again and assert `event.count` is STILL `=== 1` (no second row — RES-06 idempotency via fingerprint jobId + ON CONFLICT).
6. Teardown: `worker.close()` (afterEach), then `queue.close()`, `connection.quit()`, `prisma.deadLetterEvent.deleteMany({where:{fingerprint}})`, `prisma.event.deleteMany({where:{fingerprint}})`, `prisma.$disconnect()` (afterAll). Per-test `timeout: 20_000`.

If the API package cannot import `buildWorker`/`createCrmPolicy` from `@omnisync/worker` (no dependency), either (a) add `@omnisync/worker` as a devDependency of `apps/api` for the test, OR (b) place this integration test in `apps/worker/tests/integration/requeue.test.ts` instead and import `requeueDlqEntry` from `@omnisync/api` (add `@omnisync/api` as a worker devDep). Choose whichever keeps the dependency graph acyclic — prefer (b): put the test in the worker package (which already depends on db+queue and can add a dev-only @omnisync/api import) so the API package stays lean. If neither import direction is clean, drive the pipeline by calling `buildProcessor(...)` directly on the re-enqueued job data (Phase 3 pattern) and assert the count — the requeue SERVICE call + a direct processor invocation together prove the path.
