  <action>
Make CI actually run the 25 existing tests with an enforced coverage threshold. Four edits:

1. **turbo.json** — add a `test` task to the `tasks` object:
   ```json
   "test": { "dependsOn": ["^build"] }
   ```
   `^build` is REQUIRED because @omnisync/api tests import workspace packages (@omnisync/config,
   @omnisync/types, @omnisync/db, @omnisync/queue) that resolve to their `dist/` builds. Place it
   alongside `build`/`typecheck`. Do NOT set `outputs` (test produces no cacheable build artifact).

2. **package.json (root)** — add to `scripts`:
   ```json
   "test": "turbo run test"
   ```
   Packages without a `test` script are simply skipped by turbo — that is fine and expected.

3. **apps/api/vitest.config.ts** — add a `coverage` block inside `test`. Keep the existing
   `globals`, `environment`, `setupFiles`, `include`, `passWithNoTests`. Add:
   ```ts
   coverage: {
     provider: "v8",
     include: ["src/**"],
     exclude: ["src/index.ts", "src/types/**"],
     thresholds: { lines: 80 },
   },
   ```
   Rationale for exclusions (state in commit/summary): `src/index.ts` is the wiring entrypoint that
   opens real Redis/queue connections at import (Phase-6 integration/E2E territory, not unit-testable);
   `src/types/**` is declaration-only `.d.ts` with no executable lines. `provider: "v8"` is the default
   but stated explicitly for clarity.

   **IMPORTANT — measure before setting the threshold.** Run
   `pnpm --filter @omnisync/api test:coverage` FIRST and read the actual `% Lines` for the included set.
   Set `thresholds.lines` to 80 ONLY IF actuals pass with headroom. If actual lines coverage is
   comfortably above 80 (e.g. ≥85), you MAY also add `functions`/`branches`/`statements` thresholds at
   a value the actuals clear with margin. NEVER commit a threshold that fails. If actual line coverage
   is below 80, do NOT lower the gate — STOP and report (the included src is expected to be well-tested;
   a sub-80 number means an exclusion is wrong or a test is missing, which is a finding, not a config tweak).

4. **.github/workflows/ci.yml** — add a step AFTER the existing "Build" step (and before/after "Lint"
   is fine; place after Build):
   ```yaml
   - name: Test (with coverage gate)
     run: pnpm test -- --coverage
   ```
   turbo forwards `--coverage` to each package's `test` script (`vitest run --coverage`); the vitest
   config thresholds make the run exit non-zero below the bar, failing CI. The `^build` dep on the
   turbo test task means turbo rebuilds packages as needed, but CI already runs a "Build" step prior.

  <name>Task 3: Migrate deprecated Zod v4 string-format APIs (z.iso.datetime, z.url)</name>
  <files>packages/types/src/event.ts, packages/config/src/env.ts</files>
  <action>
Replace the two deprecated Zod v4 string-format chains with their current top-level equivalents.
Both files import from `zod/v4`, so `z.iso.datetime` and `z.url` are available without import changes.

1. **packages/types/src/event.ts** (line 9) — change:
   ```ts
   occurredAt: z.string().datetime(),
   ```
   to:
   ```ts
   occurredAt: z.iso.datetime({ offset: true }),
   ```
   The `{ offset: true }` is DELIBERATE and end-to-end-coherent, not cosmetic: a prior fix
   (commit eb6a4d9 / quick 260610-s0n) canonicalizes `occurredAt` via `new Date(occurredAt).toISOString()`
   BEFORE fingerprinting, specifically so offset variants (e.g. `+07:00`) dedupe to the same fingerprint.
   But the DEFAULT datetime validator REJECTS timezone offsets at the API boundary, which would make that
   normalization unreachable for offset inputs. Accepting offsets here + canonicalizing downstream is the
   coherent behavior. State this rationale in the commit/summary.

2. **packages/config/src/env.ts** — change all three:
   ```ts
   DATABASE_URL: z.string().url(),
   DIRECT_URL: z.string().url(),
   REDIS_URL: z.string().url(),
   ```
   to:
   ```ts
   DATABASE_URL: z.url(),
   DIRECT_URL: z.url(),
   REDIS_URL: z.url(),
   ```
   `z.url()` uses the WHATWG URL parser, which accepts any scheme — `postgresql://...` and `redis://...`
   continue to pass (verify via the build/typecheck gate; existing env-dependent tests load these URLs).

After editing the packages, you MUST run `pnpm build` (turbo rebuilds @omnisync/types and @omnisync/config
dist/) BEFORE running api typecheck/tests, so the api picks up refreshed `dist/` types. If Biome complains
